Plan to Build a Web‑based World Studio for Gaussian‑Splat Editing and Autonomous‑Robotics Simulation
1 Overview and design constraints
World Studio must be a desktop‑class world renderer, editor and simulator built on the design prototype provided in the zip file. It needs to support many use‑cases—visualising and editing large 3D Gaussian splat datasets, simulating and piloting drones, cars and indoor robots, configuring sensor rigs, and recording & replaying episodes—all within a single consistent UI. The product must reproduce the high‑quality feel of the prototype (dark theme, warm‑charcoal palette, floating panels, subtle type hierarchy) and maintain the six modes of operation described in codex.md: View, Edit, Simulate, Pilot, Sensors and Episode. The design documents state that the current canvas renderer (ws‑render.js) is a procedural stand‑in; the real product must swap it with a “real 3DGS/PLY renderer” and abide by a strict API contract (render modes, per‑splat selection, class isolation, camera frustums, agent markers, ground grid, etc.)[1]. The design further notes that destructive operations should be undoable and that the UI must persist render mode, accent colour, density and panel visibility across sessions[1].
To meet these requirements, we must select technologies that integrate well with Three.js, run in a browser or cross‑platform desktop environment, support high‑fidelity rendering of Gaussian splats and meshes, and provide real‑time physics and sensor simulation. The user explicitly requests not to use Unity; instead, we should achieve Unity‑like capabilities through open‑source web‑compatible tools. Below is an evaluation of the available repositories and a proposed multi‑phase plan to construct World Studio.
2 Evaluation of relevant repositories
2.1 Spark 2.0/2.1 (sparkjsdev/spark)
	•	Purpose. Spark is an advanced 3D Gaussian‑splatting renderer for Three.js. Its README lists features such as integration with the Three.js rendering pipeline, portable performance across devices, and support for many splat file formats (*.PLY, *.SPZ, *.SPLAT, *.KSPLAT, *.SOG)[2]. It can render multiple splat objects together with correct sorting, supports multiple viewpoints simultaneously and offers fully dynamic per‑splat editing—including colour edits, displacement, skeletal animation and GPU‑based shader graph editing[2]. Spark 2.0 introduced a level‑of‑detail (LoD) tree system that streams huge worlds by selecting a fixed number of splats per frame and allows high‑precision encoding[3]. Spark 2.1 continues this with performance enhancements. Because Spark already provides efficient LoD generation and streaming, it will remain the primary renderer and streaming solution for World Studio; other engines (e.g., Aholo or PlayCanvas) will serve only as inspiration or optional extensions.
	•	Suitability. Spark directly satisfies the renderer contract: it runs on Three.js and exposes classes like SparkRenderer and SplatMesh that can load splat files from URLs and render them within a normal Three.js scene[4]. Its dynamic editing API can be used to implement operations like per‑splat selection, transformation, SH‑degree truncation and brightness adjustment required in Edit mode. Spark’s LoD system allows interactive viewing of large scans without blocking the UI.
2.2 SplatTransform, SuperSplat Viewer & Editor
	•	SplatTransform (dimforge/splat‑transform). This CLI/library reads and writes multiple splat formats and provides operations such as merging, translation, rotation, scaling, filtering, decimation, LOD tree generation and voxel‑based collision‑mesh export[5][6]. It can output collision meshes (.OBJ), LOD trees (.SOG), voxel grids and WebP images. These tools will be invaluable for offline preprocessing of captured scans and for generating collision meshes that feed into the physics engine.
	•	SuperSplat Viewer/Editor (playcanvas/supersplat‑viewer, superspl.at/editor). The official viewer is compiled into a static web site that accepts PLY/SOG/PLYz files and exposes many options via URL parameters; it uses WebGL/WebGPU through PlayCanvas and provides reference code for camera tracks, annotated camera poses and UI settings[7][8]. The SuperSplat Editor is a browser‑based tool for inspecting, editing, optimising and publishing 3D Gaussian splats[9]. These projects demonstrate how to implement editing workflows (brush selection, deletion, cropping, merging) in a web environment and will serve as inspiration for Edit mode. However, they rely on PlayCanvas rather than Three.js, so they cannot be reused directly.
2.3 PlayCanvas engine & PCUI
The PlayCanvas engine is an open‑source WebGL/WebGPU game engine that includes advanced features such as physically based rendering (PBR), VR/AR support and built‑in physics via ammo.js (the WebAssembly port of Bullet). The README explicitly states that PlayCanvas can load and render Gaussian splats, supports advanced graphics, XR, animations and streaming, and integrates an ammo.js physics engine[10]. The PCUI library provides styled UI components and React bindings[11][12]. While PlayCanvas demonstrates a full stack for 3D editing tools (e.g., the SuperSplat viewer/editor), our world studio uses Three.js and custom design tokens, so we will not adopt PlayCanvas wholesale. Instead, we will borrow patterns for integrating ammo.js physics into a web app. If necessary, we can import ammo.js directly rather than through PlayCanvas, but we may prefer Rapier (see below).
2.4 Rapier & Salva (dimforge/rapier, dimforge/salva)
Rapier is a set of high‑performance 2D/3D physics engines written in Rust. The official site emphasises that Rapier provides multi‑platform physics simulations for games, animation and robotics, runs cross‑platform (including web), supports joint constraints for vehicles/robots, offers contact events and scene queries for collision detection and ray casting, and optionally enables SIMD and parallelism for performance[13]. Rapier has official NPM packages that compile to WebAssembly, making it suitable for browser and desktop use[14]. Deterministic mode ensures reproducibility across platforms[15]. Because our world studio runs in Three.js and needs accurate rigid‑body simulation for drones, cars and robots, Rapier is an excellent candidate.
Salva is a fluid‑simulation library with optional coupling to Rapier[16]. If fluid dynamics (e.g., for indoor water or dust) are later needed, Salva could be integrated via WebAssembly.
2.5 Bullet, MuJoCo and other physics engines
Bullet is a C++ physics SDK widely used in games and robotics. The repository states that Bullet provides real‑time collision detection and multi‑physics simulation[17]. Web integrations exist (ammo.js) but require manual binding. MuJoCo is a high‑fidelity physics simulator used in robotics research; its README emphasises interactive simulation of articulated structures with contact dynamics and interactive visualisation[18]. Both engines are powerful but either require heavy native builds (MuJoCo) or rely on WebAssembly wrappers (ammo.js). Since Rapier offers similar capabilities with first‑class JavaScript bindings, we prefer Rapier.
2.6 AI2‑THOR, AirSim, Project AirSim, Isaac Sim
	•	AI2‑THOR is an open‑source platform for embodied AI with high‑quality Unity scenes. It provides interactive environments, multi‑modality images and domain randomisation. The README shows how to initialise a controller and step actions[19]. However, AI2‑THOR relies on Unity and is not designed for web. It is valuable for dataset generation but not suited as our runtime.
	•	AirSim is Microsoft’s open‑source simulator for drones and cars, built as an Unreal Engine/Unity plugin. The README explains that AirSim is cross‑platform, supports software‑in‑the‑loop simulation with flight controllers (PX4, ArduPilot) and provides APIs to retrieve data and control vehicles programmatically[20]. It also has a Computer Vision mode for collecting depth, disparity, surface normals and segmentation without vehicles[21]. Project AirSim, maintained by IAMAI Simulations, continues AirSim’s development on Unreal Engine 5 and defines three layers: simulation libraries, a plugin connecting controllers/physics/renderers per robot type, and a client library[22]. These projects require Unreal Engine 5 or Unity 3D and thus do not fit our web‑based requirements, but they inspire features such as sensor APIs, flight control interfaces and multi‑agent support.
	•	Isaac Sim and Isaac Lab are part of NVIDIA Omniverse. Isaac Sim supports importing URDF/MJCF/CAD robots, uses GPU‑accelerated physics with high‑fidelity RTX rendering and offers workflows for synthetic data generation, reinforcement learning, ROS integration and digital‑twin simulation[23]. Isaac Lab builds a robot‑learning framework on top of Isaac Sim. Both require Omniverse, powerful GPUs and proprietary licenses, so they are not feasible for a browser‑based tool.
In summary, the heavy simulation engines (AirSim/Isaac Sim) are instructive but not adoptable; we must design our own modular physics and control system using Rapier or ammo.js to achieve similar behaviour inside Three.js.
2.7 Aholo Viewer (manycoretech/aholo‑viewer)
Purpose and capabilities. Aholo Viewer is a feature‑rich, high‑performance 3DGS rendering engine that emphasises scalability and runtime efficiency. Its home page highlights several advantages: it uses level‑of‑detail (LOD) and streaming to control memory and GPU usage, provides tiered rendering presets to balance performance and visual quality, offers GPU‑aware rendering optimisations, supports multiple file formats (ply, spz, sog, splat, lcc and ksplat) and is licensed under MIT[24][25]. Users can choose between quality‑first, performance‑first and extreme‑performance profiles to suit different devices[25].
Large‑scene LoD streaming. Aholo Viewer’s defining feature is its chunk‑level LoD streaming. The engine splits a massive 3DGS dataset into chunks using an octree and pre‑computes multiple LoD levels per chunk. At runtime, a scheduler loads chunks by view‑frustum priority and uses streamed batch fetching to make the first view available within ∼10 s, enabling smooth navigation of scenes containing hundreds of millions of Gaussians[26]. The accompanying manual describes the generation process: (1) chunk splitting via an octree with a configurable maximum number of Gaussians per chunk; (2) Gaussian search and fusion to progressively merge Gaussians and enlarge remaining splats while culling those with low opacity; and (3) output processing that packs low‑level data into a lod-meta.json file that lists all chunks and LoD levels[27]. The lod-meta.json file stores the chunk tree, bounding boxes, file indices and LoD offsets[28]. This approach allows real‑time streaming of billion‑scale 3DGS scenes in the browser[26].
Physics collision and voxel colliders. The Aholo framework includes a physics collision module that converts reconstructed 3DGS scenes into queryable colliders. It generates voxel colliders from 3DGS assets (or accepts a pre‑computed collision mesh) and uses a sparse voxel octree (SVO) layout to encode occupancy[29]. At runtime, this structure supports raycast, sphere and capsule queries for grounding, picking, character depenetration, camera obstacle avoidance and area limits[30][31]. A voxel-meta.json file and voxel.bin binary store the octree, while collision.glb provides an optional mesh[32].
Complete toolchain support. Aholo Viewer accompanies its renderer with a toolchain for conversion, LoD generation, editing, picking and voxel collider generation[33]. It integrates with @manycore/aholo-splat-transform to produce LOD trees and voxel data, and offers configuration presets for different 3DGS formats and precision targets. Cloud‑rendering support allows mixing 3DGS and high‑fidelity mesh content and streaming the composition to low‑spec devices[34].
Relevance to World Studio. Although our stack will be built around Spark and Three.js, Aholo Viewer introduces valuable concepts that we should incorporate. Its chunked LoD streaming shows how to handle billion‑scale Gaussian scenes by loading only the visible chunks and gradually refining them. Its physics collision module demonstrates how to convert a 3DGS world into a sparse voxel octree and perform real‑time ray and capsule queries for walk mode and obstacle avoidance. We will adapt these preprocessing pipelines using SplatTransform to generate lod-meta.json and voxel-meta.json files and load them into our viewer. Aholo’s tiered rendering presets and GPU‑aware optimisations also inspire user‑selectable performance profiles that trade off fidelity and speed. However, Spark remains our primary choice for rendering and streaming; Aholo’s features will be treated as complementary techniques rather than a replacement. Finally, its open‑source licence means we can study its code for reference.
3 High‑level architecture for World Studio
3.1 Core technology stack
	•	Rendering: Use Spark 2.1 as the Gaussian‑splat renderer integrated with Three.js. Set up a SparkRenderer instance inside the viewport and load splat files via SplatMesh objects[4]. For each loaded world, pre‑process the data into a chunked LoD tree: split the scene into spatial chunks and compute multiple LoD levels per chunk (via SplatTransform or Spark’s utilities) to enable streaming of billion‑scale scenes[27]. At runtime, implement a scheduler that requests visible chunks on demand and gradually refines them, similar to Aholo Viewer’s approach[26]. Use Spark’s dynamic editing API to implement per‑splat transformations, colour edits, brightness adjustments and skeletal animations[2].
	•	Physics: Integrate Rapier (via the official NPM WebAssembly packages) as the rigid‑body physics engine. Rapier supports joint constraints (for vehicles/robots), collision events and scene queries, and runs cross‑platform[13]. Use SplatTransform to export collision meshes from splat scenes, then convert them to Rapier colliders (triangular meshes or voxelised volumes). For vehicles, construct composite rigid bodies (drone frames, wheels, etc.) and configure joint constraints (e.g., wheel axles, gimbal joints). For simulation of simple props (boxes, spheres), generate Rapier shapes directly.
	•	Three.js integration: Attach the Rapier world step inside the Three.js render loop. Update object transforms in both the physics engine and Spark’s scene. Because Spark draws splats via a separate renderer, we need to sync camera parameters (position, orientation and field‑of‑view) between Three.js and Spark. Spark already supports multiple viewpoints; our UI should maintain only one primary viewport at a time.
	•	Sensors: Implement sensors by rendering the scene from additional virtual cameras and using Three.js render targets:
	•	RGB camera: Use a secondary WebGLRenderTarget; copy the colour buffer.
	•	Depth camera: Use THREE.WebGLRenderer.readRenderTargetPixels() or custom shaders to output linear depth maps. Alternatively, use Rapier’s scene queries to sample distances.
	•	Semantic camera: Keep a mapping from object IDs/classes and render an ID buffer; assign each class a unique colour index. Spark’s semantic mode palette can be reused for the colour mapping.
	•	Lidar/IMU: Use Rapier’s scene queries (ray‑casting, sweep tests) to sample ranges and collisions[35]. For IMUs, integrate acceleration and angular velocity from the physics engine.
	•	Networking and API: Expose a WebSocket or REST API to control vehicles remotely. Provide methods similar to AirSim’s API (e.g., arm, takeoff, moveByVelocity, setCarControls, reset, getImages) to integrate with training scripts or autopilot software. Use JSONRPC for cross‑language compatibility. Keep the API stateless where possible and persist simulation state inside the server.
3.2 Implementing the six modes
	•	View Mode (read‑only inspection):
	•	Use Spark to load the world from .ply or .splat files. Provide pills to switch between splat, points, mesh, semantic and depth render modes as required in codex.md[3]. For mesh mode, build a THREE.Mesh from the collision mesh produced by SplatTransform. For points mode, convert the splats to point cloud geometry.
	•	Display a world tree listing all loaded worlds and their subobjects. Use the class legend to highlight/isolate semantic classes. Implementation note: set per‑splat alpha to 18 % for non‑isolated classes to replicate the prototype.
	•	Provide a playback capsule for captured sequences; scrub through recorded frames and update the viewport and sensor feeds. Use the Episode infrastructure (see below) for actual playback.
	•	Edit Mode (Studio):
	•	Selection tools: Implement brush and rectangular selection by projecting cursor positions into world space using the viewport camera and calling Spark’s radius picking or writing a custom selection shader. In addition to Spark’s built‑in editing, call Rapier ray‑casts to check for collisions with the ground and avoid selecting occluded splats. Maintain an undo/redo stack per codex.md guidelines.
	•	Deletion and undo: Mark selected splats as deleted without removing them from GPU buffers. Use an index mask to hide them in the renderer and store operations on a history stack. Provide a “ghost deleted” toggle; when on, draw deleted splats in red with ~10 % opacity.
	•	Optimisation: Allow the user to adjust SH degree (0–3), brightness/exposure and remove outliers. Outlier removal can use SplatTransform’s filtering or our own statistical filter (e.g., z‑score on density). Live updates to the LoD tree should be triggered after any change.
	•	Export & publish: Provide buttons to export the cleaned world to .PLY, .SPLAT or .SOG. Use SplatTransform to write the file and show estimated file sizes. Optionally integrate with PlayCanvas or Cloud storage for hosting.
	•	Simulate Mode:
	•	Split the screen into left and right panels (50/50) as defined in codex.md[3]. The left panel displays the live sensor feed (RGB, depth, segmentation, lidar, IMU), while the right panel overlays the metric reconstruction (ground truth) using the same world data. Provide a frames list and bottom cards summarising session status (e.g., synchronisation error, dropped frames) and agent statuses. Use Rapier to update the physical simulation each frame and compute metric reconstruction (e.g., by ray‑casting to measure errors between perceived range and world model).
	•	Pilot Mode:
	•	Provide WASD keyboard controls for driving the selected agent (drone, car or robot). Use Rapier to integrate forces/torques and update the agent’s pose. Display a PiP (picture‑in‑picture) view showing the agent’s first‑person camera. Show a spawn‑props panel to drop obstacles (boxes, spheres) into the world; these objects become Rapier rigid bodies with colliders. Draw trajectory breadcrumbs and step counters. Provide start/stop buttons and ensure controls disable when no agent is selected.
	•	Sensors Mode:
	•	Let users build sensor rigs by placing camera/lidar/IMU frustums in the world. Use the viewport to draw 3D frustum gizmos and labels; update extrinsic/intrinsic parameters via a panel. Use toggle switches to enable/disable channels (RGB, depth, segmentation, point cloud). Provide a preview card that shows each channel’s output in real time. Save rig configurations to JSON and allow import/export.
	•	Episode Mode:
	•	Record the sequence of agent actions, object events and sensor captures during simulation. Represent this as a multi‑track timeline (agent actions, object events, capture ticks) exactly like the prototype’s track panel. Provide play/pause controls and a playhead; scrubbing the timeline updates the simulation deterministically because Rapier’s determinism ensures reproducible stepping[15].
	•	Support exporting episodes (with sensor data, actions and metadata) for training datasets or offline analysis.
3.3 Miscellaneous features and considerations
	•	Collision Mesh & Voxel Collider Generation: Use SplatTransform to generate either a triangle‑mesh collision model (.OBJ) or a voxel grid from the Gaussian splats[5]. When producing voxel data, adopt the sparse voxel octree (SVO) layout used in Aholo Viewer: encode occupancy in an octree with separate nodes, mixed‑leaf masks and solid leaf markers[31]. Export the octree to voxel-meta.json and voxel.bin for consumption at runtime[32]. At runtime, load this data and implement raycast, sphere and capsule queries for picking, grounding and character depenetration[30]. For physical simulation, convert the mesh or voxel hull into Rapier colliders. For large scenes, pre‑compute simplified collision meshes or coarse voxels to maintain performance.
	•	Level‑of‑Detail streaming: Large reconstructed worlds can contain hundreds of millions of Gaussians, far beyond the GPU memory of most devices. Following lessons from Aholo Viewer, preprocess each world to build a chunked LoD tree using SplatTransform or Spark’s utilities. Split the scene into spatial chunks (octree) with a configurable maximum number of Gaussians and fuse Gaussians within each chunk to generate multiple LoD levels[27]. Export a lod-meta.json describing the chunk tree, LoD levels and file offsets[28]. In the viewer, implement a scheduler that loads chunks by view‑frustum priority and gradually refines them; this ensures a usable first view within seconds and scales to billion‑scale scenes[26]. Provide UI controls for density and performance profiles (quality‑first versus performance‑first) and persist these preferences in localStorage.
	•	State persistence: Use localStorage to remember the last selected mode, render mode, accent colour, density and panel style. On start‑up, apply these preferences automatically, as indicated in codex.md.
	•	Undo/Redo & History: Implement a history stack storing command objects (select, delete, invert, transform). Each command must support do() and undo() functions. Use React context to expose this stack across panels.
	•	Accessibility & performance: Because the UI is heavy, enable a “dense” mode that reduces padding and font sizes; this corresponds to the .dense variant in ws‑styles.css[3]. Provide a “docked” panel style for users who prefer solid panels without blur. Use requestAnimationFrame to synchronise the physics step and rendering; if physics updates fall behind, decimate the LoD further.
	•	Cross‑platform packaging: Target macOS first (as per the design doc), but plan for Windows and Linux using Electron or Tauri. Electron can host the web‑based React/Three.js app and embed a full‑screen canvas; panels remain HTML overlays with CSS styling. When packaging, include the Google Fonts (IBM Plex Mono, Spline Sans, Source Serif 4) and ensure the window is borderless. Provide an option to run headless (no GUI) to produce data sets or train RL agents, similar to AirSim’s off‑screen rendering options[36].
4 Roadmap for development
	•	Prototype integration (Month 1–2).
	•	Build the React app skeleton based on ws-app.jsx, ws-panels.jsx and the design system. Ensure the UI scales to 1920×1080 and implements all design tokens.
	•	Integrate Three.js and Spark 2.1. Load the provided data/loft_04/gaussians.ply and verify that Spark reproduces all five render modes (splat/points/mesh/semantic/depth) with the correct colour palette and ground grid. Replace the stand‑in ws-render.js API with calls into Spark.
	•	Use SplatTransform to convert the sample dataset into a collision mesh; integrate this with Rapier and verify basic rigid‑body collisions (dropping a box onto the floor).
	•	Edit mode & LoD streaming (Month 2–3).
	•	Implement selection and deletion tools using Spark’s per‑splat editing capabilities. Add history stacks and an “undo” button.
	•	Expose controls for SH degree, brightness/exposure and outlier removal; implement filters using SplatTransform or custom algorithms.
	•	Add export functionality to output cleaned splats in different formats.
	•	Preprocess large datasets into LoD trees and support dynamic density adjustment.
	•	Physics & Agents (Month 3–4).
	•	Integrate Rapier with the Three.js scene. Create classes for drone, car and generic robot. Each agent should support multiple degrees of freedom, motor controllers and sensors.
	•	Implement user‑controllable agent driving (WASD) and autopilot controllers (PID loops, trajectory following).
	•	Map Rapier’s joint constraints to vehicle structures (e.g., wheel suspensions, gimbals) and verify realistic dynamics.
	•	Provide an API for remote control via WebSocket/JSONRPC. Use this to implement simple RL scripts for navigation tasks.
	•	Sensors & Simulate mode (Month 4–5).
	•	Implement sensor rigs: UI for placing and configuring cameras, lidar and IMU. Integrate sensor simulation with the physics engine and rendering pipeline.
	•	Build Simulate mode with side‑by‑side live feed vs. metric reconstruction. Include timeline playback, session status cards and error metrics.
	•	Pilot & Episode modes (Month 5–6).
	•	Build the Pilot mode UI: spawn props, drive vehicles, PiP view, trajectory breadcrumbs and step counter.
	•	Implement Episode mode: recording of actions/events, multi‑track timeline with a playhead, scrubbing and replay. Ensure determinism by stepping Rapier with fixed time‑steps and saving seeds for the random number generator.
	•	Polish and packaging (Month 6+).
	•	Add accent‑colour switching, density toggles and docked panel style.
	•	Optimise performance: cull invisible splats, simplify collision meshes, multi‑thread heavy computations via Web Workers.
	•	Package the application via Electron or Tauri; implement auto‑updates and crash reporting.
	•	Prepare documentation and sample worlds; provide developer APIs and examples.
5 Conclusion
By combining Spark 2.1 for high‑fidelity Gaussian‑splat rendering, Rapier for cross‑platform physics simulation, SplatTransform for offline editing and collision‑mesh generation, and the detailed UI specifications from the World Studio design repo, we can construct a modern, web‑based world simulator that rivals Unity‑quality tools. The plan above details how to implement each mode of the prototype, integrate physics and sensors, and support drones, cars and robots. Heavyweight simulators like AirSim and Isaac Sim informed our feature set but are unsuitable for our browser‑based environment. Instead, our solution uses open, web‑friendly technologies to create a flexible, extensible platform for real‑time simulation and embodied‑AI research.
